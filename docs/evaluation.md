# 代打AI — テスト・評価フレームワーク

## 概要

LT自動化パイプライン（テーマ → アウトライン → スライド仕様JSON → PPTX → 音声 → スライドショー）に対するテストと品質評価のフレームワーク。

### テストと評価の区別

本ドキュメントでは、以下の2つを明確に区別する:

| | ガードレール（テスト） | 品質評価（Evaluation） |
|---|---|---|
| **性質** | 決定論的。入力が同じなら結果は同じ | 非決定論的。LLM出力に依存し、実行ごとに結果が異なる |
| **判定** | Pass / Fail（二値） | スコア / 傾向（連続値） |
| **目的** | 壊れたものを出荷しない（ガードレール） | 品質の傾向を把握し改善する |
| **実行** | CI自動実行、違反時はパイプラインを停止 | 定期的に実施、トレンドを追跡 |
| **対象** | コード・データ構造・ファイル形式 | LLM生成コンテンツ・知覚的品質 |

### 設計方針

- **ガードレールはCI必須**: 構造的な正しさは自動テストで保証する
- **評価は小さく始める**: 問題が顕在化した軸を重点的に拡充する
- **混同しない**: テストが通っても品質が高いとは限らない。評価スコアが低くてもテストは通る

# Part A: ガードレール（テスト）

決定論的に検証可能な制約。CIに組み込み、違反時はパイプラインを停止する。

## A1. SlideSpec構造バリデーション

パイプラインの中間生成物であるSlideSpecが、構造的な制約を満たしているかを検証する。
LLMが生成するSlideSpec JSONの妥当性を、下流（PPTX生成）に渡す前にチェックするガードレール。

### 実装済み（バリデーション・テストあり）

| テスト項目 | アサーション | 根拠 | テスト箇所 |
|-----------|-------------|------|-----------|
| レイアウト種別 | `layout`が定義済みの値のいずれか | python-pptxのレイアウトインデックス参照に必須 | `test_enrich_outline.py` (`slide_spec.py`のVALID_LAYOUTS) |
| タイトル存在 | blank以外のスライドに`title`が非空 | レイアウト描画に必須 | `test_enrich_outline.py` (TestA1ガードレール_タイトル非空) |
| スライド枚数 | 1 ≤ len(slides) ≤ max_slides | 登壇形式に応じた物理的制約 | `test_enrich_outline.py` (TestA1ガードレール_スライド枚数) |
| ノート必須 | title_slide/section_header以外に`note`が非空 | 音声合成の入力に必須 | `test_enrich_outline.py` (TestA1ガードレール_ノート必須) |
| 情報密度 | コンテンツスライドの箇条書き数が1〜8の範囲内 | 0項目=空スライド、9項目以上=視認不可 | `test_enrich_outline.py` (TestA1ガードレール_情報密度) |
| 推定発話時間 | 全ノートの合計文字数 / 5.0 ≤ max_talk_duration_sec | 持ち時間制限 | `test_enrich_outline.py` (TestA1ガードレール_推定発話時間) |

`validate_slide_spec()` は `max_slides` と `max_talk_duration_sec` をキーワード引数で受け取り、登壇形式に応じて上限を変更できる。
デフォルト値は `slide_spec.py` に定義: `MAX_SLIDES=20`, `MAX_TALK_DURATION_SEC=300`（5分LT想定）。
テストは `@pytest.mark.parametrize` で5分LT / 15分LT / 30分講演の3形式をカバー。

`enrich_outline.py` は `--duration` 引数で持ち時間（分）を受け取り、`max_slides = duration × 4`, `max_talk_duration_sec = duration × 60` を自動算出する。

## A2. PPTX構造バリデーション

生成されたPPTXファイルが、アプリケーションで開ける最低限の構造的正しさを持つかを検証する。

### 実装済み（テストあり）

| テスト項目 | アサーション | テスト箇所 |
|-----------|-------------|-----------|
| ファイル生成 | 出力パスにPPTXファイルが存在する | `test_slide_builder.py` (`test_PPTXファイルとして保存できる`) |
| python-pptxで開ける | `Presentation(path)`が例外を投げない | `test_slide_builder.py` (`test_保存したPPTXを再度開ける`) |

| 改行混在 | 空行が混在しても正しく解析 | `test_outline_parser.py` |
| 空ノート | 空文字列ノートでスキップ | `test_synthesize_audio.py` |

### 実装済み: 下流パイプライン（PPTX生成）のエッジケース耐性

| テスト項目 | アサーション | テスト箇所 |
|-----------|-------------|-----------|
| 特殊文字タイトルでPPTX生成 | 絵文字・RTL・HTML・NFD・NULL・ゼロ幅の6パターンでクラッシュしない | `test_slide_builder.py` (TestA3下流パイプラインエッジケース) |

外部APIをモックして障害を注入し、エラーハンドリング動作を検証する。

> **アーキテクチャ上の注意**: 画像生成（Gemini API）とSVG生成は `skills/daida-ai/scripts/` 配下の
> ワークフロースクリプトが担当し、コアライブラリ（`daida_ai/lib/`）は生成済みの画像パスを受け取るだけ。
> そのため、Gemini→SVGの自動フォールバックはSkillワークフロー側の責務であり、
> コアライブラリのユニットテストではカバーしない。

| 障害箇所 | アサーション | テスト箇所 |
|----------|-------------|-----------|

| TTS API障害（全失敗） | 全スライドNoneを返し、クラッシュしない | `test_synthesize_audio.py` |

### 計画中（テスト未実装）

| 障害箇所 | アサーション |
|----------|-------------|
| Skillワークフロー: Gemini障害→SVGフォールバック | SVGで代替画像を生成し、パイプライン完走する |

> **TTS障害時の復旧フロー**: TTS失敗時は `synthesize_notes()` が失敗スライドをスキップし、成功分のみ返す（部分成功）。
> `synthesize_audio.py` は失敗数をstderrに報告して正常終了する。
> TTS復旧後にStep 4を再実行すれば全スライドの音声が生成される。

> **LLM API障害について**: パイプライン全体がClaude Code上で実行され、
> LLMとの対話はClaude Code自身が担当する（`daida_ai/lib/` も `scripts/` もLLM APIを直接呼ばない）。
> LLM API障害時はClaude Code自体が動作不能になるため、ライブラリ側でのテストは不要・不可能。
> LLMが生成する中間データの妥当性はA1（SlideSpec構造バリデーション）でガードされている。

## A5. タイミング推定式の精度検証

推定式 `len(note.strip()) / 5.0 * 1000`（1秒あたり5文字）の精度を検証する。

### 実装済み（テストあり）

推定式の基本動作は `slideshow.py` の `_estimate_note_duration_ms()` で実装済み。

| テスト項目 | アサーション | テスト箇所 |
|-----------|-------------|-----------|
| ノートありスライドの推定時間 | ノート長に応じたadvTmが設定される | `test_slideshow.py` (`test_ノートありスライドのadvTmがsilent_durationより長い`) |
| 長いノートの推定時間 | 長いノートほど長いadvTmになる | `test_slideshow.py` (`test_長いノートは長いadvTmになる`) |
| 最低値制約 | 短いノートでも最低3秒（`_MIN_NOTE_DURATION_MS`） | `test_slideshow.py` (`test_ノートベースタイミングの最低値は3秒`) |

| MP3デュレーション計測（MPEG1 128kbps） | 既知データサイズから期待ms値と一致 | `test_slideshow.py` (TestA5_estimate_mp3_duration_ms) |
| MP3デュレーション計測（MPEG2 64kbps） | 既知データサイズから期待ms値と一致 | `test_slideshow.py` |
| ID3タグ付きMP3 | ID3スキップ後のデータ量で正しく計算 | `test_slideshow.py` |
| フレームヘッダなし | 0を返す | `test_slideshow.py` |
| 空データ / 10バイト未満 | 0を返す | `test_slideshow.py` |

テスト用MP3データは `conftest.py` の `build_mp3_frame()` / `build_id3_header()` で構築。fixture名: `mp3_mpeg1_128kbps`, `mp3_mpeg2_64kbps`, `mp3_with_id3_tag`, `mp3_no_frame_header`。

### 計画中（テスト未実装）

| テスト項目 | アサーション |
|-----------|-------------|
| 実TTS音声長との比較 | 推定時間と実VOICEVOX音声長の誤差率平均 < 20% |

> **注意**: 実TTS音声長との比較はVOICEVOXがローカル起動済みの環境でのみ実行可能。
> TTS（VOICEVOX）は同一テキストに対してほぼ同一の音声長を返すため、決定論的に扱える。

### 精度検証スクリプト（参考）

```python
from pathlib import Path
from pydub import AudioSegment

def measure_timing_accuracy(audio_dir: Path, spec: SlideSpec) -> list[dict]:
    """ノートベース推定時間 vs 実音声長の比較"""
    results = []
    for i, slide in enumerate(spec.slides):
        audio_path = audio_dir / f"slide_{i:03d}.mp3"
        if not audio_path.exists() or not slide.note:
            continue
        actual_ms = len(AudioSegment.from_mp3(str(audio_path)))
        estimated_ms = int(len(slide.note.strip()) / 5.0 * 1000)
        error_rate = abs(actual_ms - estimated_ms) / actual_ms
        results.append({
            "slide": i,
            "actual_ms": actual_ms,
            "estimated_ms": estimated_ms,
            "error_rate": round(error_rate, 3),
        })
    return results
```

### 推定式のキャリブレーション

誤差率が目標を超えた場合:

1. 実測データから `文字数 / 実秒数` の平均を算出
2. 係数（現在5.0）を更新
3. テストを再実行して誤差率が改善したことを確認

# Part B: 品質評価（Evaluation）

LLMの非決定論的出力に依存する品質指標。定期的に実施し、トレンドを追跡する。
テスト（Part A）が全て通った上で、「品質が十分か」を判断するために使う。

## B1. コンテンツ品質

| 評価軸 | 指標 | 測定方法 | 目標値 |
|--------|------|----------|--------|
| アウトライン構造 | セクション数・流れの自然さ | 人間評価（5段階） | ≥ 4.0 |
| トークスクリプト品質 | 自然さ・話し言葉らしさ | 人間評価（5段階） | ≥ 4.0 |
| 一貫性 | タイトル↔本文↔ノート間の矛盾 | 人間目視確認 | 矛盾なし |

> これらはLLMが生成するコンテンツの「意味的な品質」であり、
> 構造バリデーション（A1）をパスしたSpecに対して実施する。
> A1は構造的な正しさ（タイトル非空、レイアウト種別の妥当性、将来的には情報密度等）を保証するが、
> B1は「その内容が適切か」を判断する。

### LLM-as-Judge（将来検討）

> **注意**: LLM-as-Judgeは現時点では補助的な参考情報として扱う。
> 本格導入には以下のキャリブレーションが必須:
>
> 1. 人間評価20件以上のデータセットを作成
> 2. LLM-as-Judgeの評価結果と人間評価のスピアマン相関を算出
> 3. 相関係数 ≥ 0.7 を確認してから自動評価パイプラインに組み込む
> 4. 生成に使用したモデルとは異なるモデルファミリーで評価する

導入時の評価プロンプト（参考）:

```
以下のLTスライド仕様を評価してください。

評価基準:
1. アウトライン構造（1-5）: セクションの流れは自然か、過不足はないか
2. 情報密度（1-5）: 1スライドあたりの情報量は適切か（多すぎ/少なすぎないか）
3. トークスクリプト（1-5）: 話し言葉として自然か、スライドの内容と整合しているか
4. 一貫性（1-5）: タイトル・本文・ノート間に矛盾はないか

JSON形式で回答: {"structure": N, "density": N, "script": N, "consistency": N, "issues": ["..."]}
```

## B2. プレゼンテーション品質

| 評価軸 | 指標 | 測定方法 | 目標値 |
|--------|------|----------|--------|
| 視覚的品質 | テンプレート適用・レイアウト崩れ | スクリーンショット目視確認 | 崩れなし |
| 画像の適切さ | スライド内容と画像の関連性 | 人間評価（5段階） | ≥ 3.5 |
| 音声品質 | TTS発話の自然さ・速度 | 開発者による聴取確認 | 明らかな不自然さなし |

> **視覚的品質**: リファレンス画像との自動差分比較はテンプレート変更のたびにリファレンス更新が必要で、
> メンテナンスコストが高い。リリース前のスモークテスト（PowerPointで開いて目視確認）で代替する。

> **音声品質**: 正式なMOS評価には複数の被験者と統制環境が必要。
> 個人プロジェクト規模では、開発者自身が「明らかな不自然さ（発音崩れ・異常な間）がないか」を確認する。

## B3. E2Eパイプライン評価

| 評価軸 | 指標 | 測定方法 | 目標値 |
|--------|------|----------|--------|
| E2E成功率 | テーマ入力→最終PPTX生成の完走率 | テーマセットで複数回実行 | ≥ 90% |
| 所要時間 | テーマ入力→最終成果物までの総時間 | 壁時計時間（API待ち含む） | < 5分（※条件下記） |
| 人的介入率 | 手動修正なしで使える割合 | ルーブリック（下記）に基づく分類 | ≥ 70% |

> **テスト（Part A）との違い**:
> A2はPPTXが「構造的に正しい」ことを保証するが、
> B3はパイプラインが「LLMの出力を含めて安定して動く」ことを統計的に見る。
> LLMが予想外の形式で返答した場合などの、テストで網羅しきれないケースを検出する。

#### 所要時間の測定条件

- ネットワーク: 有線LAN / 安定した回線（100Mbps以上）
- LLMモデル: Claude Sonnet以上
- TTS: VOICEVOX（ローカル起動済み）
- 画像生成: SVGフォールバック（API不使用時）

### テーマセット

カテゴリ代表を**各1つ、計5テーマ**で構成する。テーマは固定し、リグレッション検出に使う。

| カテゴリ | テーマ |
|----------|--------|
| 技術系 | 「React Server Componentsの実践」 |
| ビジネス系 | 「Q3売上報告」 |
| カジュアル | 「推し活の始め方」 |
| 短いテーマ | 「Git」 |
| 長いテーマ | 「2026年における生成AIを活用したソフトウェア開発プロセスの効率化とチーム導入戦略について」 |

> **補足**: テーマ数が5件の場合、E2E成功率90%の目標に対して1件の失敗で80%に落ちる。
> あくまでリグレッション検出が目的であり、統計的な精度保証ではない点に注意。

### 人的介入率のルーブリック

| 判定 | 基準 |
|------|------|
| **そのまま使える** | 以下の5項目すべてを満たす |
| **軽微な修正** | 1〜2項目が不合格 |
| **大幅修正が必要** | 3項目以上が不合格 |

**チェック項目**:

1. **レイアウト崩れなし**: テキストがはみ出していない、画像が重なっていない
2. **内容の論理的整合性**: スライドの流れが自然で、タイトルと本文が対応している
3. **トークスクリプトの整合**: ノートの内容がスライド本文と矛盾しない
4. **音声再生可能**: PowerPointでスライドショー再生時に音声が再生される
5. **時間内に収まる**: 全スライドの推定発話時間がLT制限時間（5分）以内

## B4. 再現性

| 評価軸 | 指標 | 測定方法 | 目標値 |
|--------|------|----------|--------|
| 再現性 | 同一テーマでの出力安定性 | 同じテーマで3回実行→構造比較 | 3回中2回以上で構造一致 |

### 構造一致の判定基準

同一テーマで3回実行した結果について、以下を比較する:

- **スライド枚数**: 全実行で同一であること
- **セクション構成**: セクションヘッダーのタイトルが概ね一致（語順・表現の揺れは許容）
- **レイアウト種別の列**: 各スライドのlayout種別が一致

> これはLLMの非決定論的特性そのものを測る指標であり、
> 「テスト」ではなく「現状把握」として位置づける。
> スコアが低い場合の対応は、プロンプト改善やtemperature調整であり、コード修正ではない。

# Part C: 技術的互換性（手動スモークテスト）

### サポートレベル

| レベル | 対象アプリ | 方針 |
|--------|-----------|------|
| **Tier 1（必須）** | PowerPoint (Windows), PowerPoint (Mac) | 全機能が動作すること。リリース前に必ず確認 |
| **Tier 2（ベストエフォート）** | LibreOffice Impress | 開閉・表示が可能であること。既知の制限は許容 |
| **Tier 3（非サポート）** | Google Slides | PPTXインポート互換性はGoogle側の実装依存。動作確認は行わない |

### 既知の制限事項

- **LibreOffice Impress**: 自動ページ送り（advTm）が動作しない（[Bug 101527](https://bugs.documentfoundation.org/show_bug.cgi?id=101527)）
- **Google Slides**: PPTXインポート時に音声・アニメーションが失われる可能性がある

### テストマトリクス（Tier 1 + Tier 2）

| 機能 | PowerPoint (Win) | PowerPoint (Mac) | LibreOffice |
|------|:-:|:-:|:-:|
| 基本開閉 | 必須 | 必須 | 必須 |
| テンプレート背景 | 必須 | 必須 | 確認 |
| 画像表示 | 必須 | 必須 | 確認 |
| 音声再生 | 必須 | 必須 | 確認 |
| 自動ページ送り | 必須 | 必須 | N/A（非対応） |
| ノート表示 | 必須 | 必須 | 確認 |

> **実施タイミング**: リリース前のスモークテスト時に、5テーマのうち1つをTier 1アプリで確認する。
> 全テーマ × 全アプリの網羅確認は不要。

> **テストとの境界**: PPTXの構造的正しさ（ZIP形式、python-pptxで開ける）はPart A2でテスト済み。
> Part Cはアプリケーション固有のレンダリング差異を確認するものであり、自動化が困難なため手動とする。
